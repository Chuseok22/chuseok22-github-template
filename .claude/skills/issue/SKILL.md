# Issue Mode

당신은 GitHub 이슈 작성 및 생성 전문가입니다.  
사용자의 대략적인 설명을 받아 GitHub 이슈 템플릿에 맞는 **제목과 본문을 작성**하고, 프로젝트 루트 경로의 `.issue/` 폴더에 저장한 뒤, 실제 GitHub Repository에 이슈를 생성합니다.

---

## 절대 금지 사항

- ❌ 이슈 본문에 코드적인 내용 작성 금지
    - 구현 방법
    - 코드 예시
    - 함수명 중심 설명
    - 파일 내부 구현 방식
    - 명령어 중심 해결 방법
- ❌ 긴급 태그를 임의로 추가하지 않음
    - 사용자가 직접 "긴급"이라고 말한 경우에만 `[긴급]` 사용
- ❌ GitHub Assignee를 임의의 다른 사용자로 지정하지 않음
- ❌ GitHub Label을 새로 생성하지 않음
- ❌ Repository에 존재하지 않는 Label을 임의로 추가하지 않음
- ❌ 실제 GitHub 이슈 생성 실패 시 성공한 것처럼 말하지 않음

---

## 사용자 입력

$ARGUMENTS

---

# 동작 프로세스

## 1단계: 프로젝트 Root 및 Repository 확인

현재 작업 중인 프로젝트의 root 경로를 기준으로 GitHub Repository 정보를 확인합니다.

### Repository 판단 규칙

- 프로젝트 root의 Git 정보를 기준으로 Repository를 추출합니다.
- 원격 Repository 정보가 있는 경우 해당 Repository를 사용합니다.
- Repository 정보는 `owner/repository` 형식으로 판단합니다.
- Repository를 특정할 수 없는 경우:
    - `.issue/` 파일은 생성합니다.
    - GitHub 이슈 생성은 진행하지 않습니다.
    - 사용자에게 Repository 정보를 요청합니다.

---

## 2단계: 이슈 타입 자동 판단

사용자 입력의 성격을 분석하여 아래 타입 중 하나로 자동 지정합니다.

| 타입 | 판단 기준 | 템플릿 |
|---|---|---|
| 버그 | 안 됨, 에러, 깨짐, 오류, 크래시, 문제, 장애, 실패, 로그, 재현 | `bug_report` |
| 기능추가 | 완전히 새로운 기능, 새 기능 추가, 신규 기능 필요 | `feature_request` |
| 기능개선 | 기존 기능 수정, 기존 동작 개선, 불편함 개선, UX 개선 | `feature_request` |
| 기능요청 | 제안, 요청, 검토 요청, 있으면 좋겠음 | `feature_request` |
| 디자인 | 디자인, UI, UX, 폰트, 색상, 레이아웃, 화면, 아이콘 | `design_request` |
| 시험요청 | 테스트, QA, 시험, 검증, 확인 요청 | `qa_request` |

### 타입 판단 우선순위

- 장애나 오류 현상이 명확하면 `버그`로 판단합니다.
- 기존 기능을 더 좋게 바꾸는 내용이면 `기능개선`으로 판단합니다.
- 기존에 없는 기능을 새로 만드는 내용이면 `기능추가`로 판단합니다.
- 아이디어나 제안 수준이면 `기능요청`으로 판단합니다.
- 화면, 색상, 배치, 문구, 시각 요소 중심이면 `디자인`으로 판단합니다.
- 검증 요청이나 테스트 요청이면 `시험요청`으로 판단합니다.

---

## 3단계: 제목 형식 결정

제목은 이슈 성격에 따라 아래 형식을 따릅니다.

## 일반 이슈 제목

긴급 이슈가 아닌 경우 대괄호 `[]`는 반드시 2개만 사용합니다.

```markdown
이모지[타입][카테고리] 설명
```

예시:

```markdown
❗[버그][로그인] 카카오 로그인 실패 문제
⚙️[기능추가][결제] 쿠폰 적용 기능 추가
🚀[기능개선][랭킹] 닉네임 표시 길이 개선
🔧[기능요청][알림] 주문 상태 변경 알림 요청
🎨[디자인][홈] 메인 배너 레이아웃 수정
🔍[시험요청][결제] 포인트 적립 QA 요청
```

## 긴급 이슈 제목

사용자가 직접 "긴급"이라고 말한 경우에만 대괄호 `[]` 3개를 허용합니다.

```markdown
이모지[긴급][타입][카테고리] 설명
```

예시:

```markdown
❗[긴급][버그][로그인] 카카오 로그인 실패 문제
🔧[긴급][기능요청][admin] 관리자 권한 확인 요청
🚀[긴급][기능개선][결제] 포인트 적립 정책 개선
```

### 제목 작성 규칙

- 긴급이 아닌 경우: 대괄호 2개만 사용합니다.
- 긴급인 경우: 대괄호 3개만 사용합니다.
- `[긴급]`은 사용자가 직접 긴급이라고 말한 경우에만 사용합니다.
- `[긴급]`은 항상 첫 번째 대괄호로 작성합니다.
- 두 번째 대괄호는 이슈 타입입니다.
- 마지막 대괄호는 카테고리입니다.
- 여러 카테고리가 필요한 경우 `/`로 묶습니다.
    - 예: `[Flutter/빌드]`
    - 예: `[결제/포인트]`
    - 예: `[admin/권한]`

---

## 4단계: 카테고리 자동 판단

사용자 입력을 분석하여 카테고리를 자동 지정합니다.

예시:

| 입력 내용 | 카테고리 예시 |
|---|---|
| 로그인, 회원가입, 인증 | `인증` 또는 `로그인` |
| 결제, 주문, 포인트, 쿠폰 | `결제` |
| 관리자, 어드민, admin | `admin` |
| 배포, 빌드, CI, CD | `CICD` 또는 `빌드` |
| 앱, Flutter, Android, iOS | `앱` 또는 `Flutter` |
| 화면, 버튼, 모달, 레이아웃 | `UI` |
| 테스트, QA, 검증 | `QA` |

카테고리를 특정할 수 없는 경우에는 사용자에게 짧게 질문합니다.

---

## 5단계: 부족한 정보 질문

아래 상황에서는 이슈를 바로 생성하지 말고 사용자에게 짧게 질문합니다.

### 반드시 질문해야 하는 경우

- 이슈 타입 판단이 불가능한 경우
- 카테고리 판단이 불가능한 경우
- 사용자의 설명만으로 이슈 제목을 만들 수 없는 경우

### 질문하지 않는 경우

- GitHub Assignee가 없는 경우
- 환경 정보가 없는 경우
- 참고 자료가 없는 경우
- 재현 단계가 일부 부족한 경우
- Label을 특정하기 어려운 경우
- Repository Label 중 적절한 Label이 없는 경우

---

## 6단계: GitHub Assignee 설정

GitHub Assignee는 항상 Claude Code를 실행하는 현재 사용자를 기준으로 설정합니다.

### Assignee 설정 규칙

- 현재 GitHub 인증 사용자를 확인할 수 있으면 해당 사용자를 Assignee로 지정합니다.
- 현재 GitHub 인증 사용자를 확인할 수 없으면 Assignee를 지정하지 않습니다.
- Assignee가 없다는 이유로 사용자에게 질문하지 않습니다.
- 다른 사람을 임의로 Assignee로 지정하지 않습니다.
- 사용자가 명시적으로 특정 GitHub username을 지정한 경우에만 해당 사용자를 Assignee로 사용할 수 있습니다.

---

## 7단계: GitHub Label 조회 및 설정

이슈를 생성하기 전에 해당 Repository에 존재하는 Label 목록을 조회합니다.

### Label 설정 규칙

- Repository에 이미 존재하는 Label만 사용합니다.
- 새로운 Label은 절대 생성하지 않습니다.
- 이슈 타입과 카테고리를 기준으로 적절한 Label을 자동 판단합니다.
- 적절한 Label이 없으면 Label 없이 이슈를 생성합니다.
- Label이 없다는 이유로 사용자에게 질문하지 않습니다.
- 존재하지 않는 Label 이름을 임의로 넣지 않습니다.

### Label 판단 예시

| 이슈 타입 | 우선 고려 Label |
|---|---|
| 버그 | `bug`, `버그`, `fix` |
| 기능추가 | `feature`, `enhancement`, `기능추가` |
| 기능개선 | `enhancement`, `improvement`, `기능개선` |
| 기능요청 | `request`, `feature request`, `기능요청` |
| 디자인 | `design`, `ui`, `ux`, `디자인` |
| 시험요청 | `qa`, `test`, `testing`, `시험요청` |

카테고리와 일치하는 Label이 Repository에 존재하면 함께 사용할 수 있습니다.

예시:

- `admin`
- `frontend`
- `backend`
- `app`
- `payment`
- `cicd`

단, 위 Label이 Repository에 실제로 존재하지 않으면 사용하지 않습니다.

---

## 8단계: 이슈 파일 저장

판단된 타입에 맞는 템플릿을 사용하여 프로젝트 root 경로의 `.issue/` 폴더에 마크다운 파일로 저장합니다.

### 저장 위치

```markdown
프로젝트_ROOT/.issue/
```

### `.issue/` 폴더 처리

- 프로젝트 root에 `.issue/` 폴더가 없으면 자동 생성합니다.
- `.gitignore`에 `/.issue`가 없으면 추가합니다.
- 이미 등록되어 있으면 중복 추가하지 않습니다.
- GitHub 이슈 생성 후에도 `.issue` 파일은 삭제하지 않고 유지합니다.

### 파일명 생성 규칙

```markdown
#[YYYYMMDD]_[순번]_[이슈타입]_[간단한설명].md
```

### 파일명 규칙 상세

- `#` 접두사는 유지합니다.
- 날짜는 현재 날짜를 `YYYYMMDD` 형식으로 사용합니다.
- 순번은 같은 날짜 내 `.issue/` 폴더의 기존 파일 수를 기준으로 3자리 자동 증가합니다.
    - 예: `001`, `002`, `003`
- 이슈 타입은 아래 중 하나를 사용합니다.
    - `버그`
    - `기능추가`
    - `기능개선`
    - `기능요청`
    - `디자인`
    - `시험요청`
- 설명은 제목에서 핵심 단어만 추출합니다.
- 공백은 언더스코어 `_`로 변경합니다.
- 파일명에 사용할 수 없는 특수문자는 제거합니다.
- `[긴급]` 여부는 파일명 이슈 타입 앞에 붙일 수 있습니다.

### 파일명 예시

```markdown
#20260520_001_버그_카카오_로그인_실패.md
#20260520_002_기능추가_쿠폰_적용_기능.md
#20260520_003_기능개선_닉네임_표시_길이.md
#20260520_004_디자인_홈_배너_레이아웃.md
#20260520_005_시험요청_포인트_적립_QA.md
#20260520_006_긴급_버그_Android_빌드_실패.md
```

---

## 9단계: 파일 내용 생성

파일 내용은 아래 구조를 따릅니다.

```markdown
{GitHub 이슈 제목}

{GitHub 이슈 본문}
```

주의:

- 첫 줄은 GitHub 이슈 제목입니다.
- 첫 줄에는 마크다운 제목 기호 `#`를 붙이지 않습니다.
- 한 줄 비운 뒤 본문을 작성합니다.
- 그대로 복사해서 GitHub 이슈에 붙여넣을 수 있어야 합니다.

## GitHub 이슈 번호 기록 (생성 성공 시)

GitHub 이슈 생성 성공 후, `.issue/` 파일 맨 앞에 아래 한 줄을 추가합니다.

형식: `<!-- GitHub Issue: #{번호} | {이슈 URL} -->`

예시:

    <!-- GitHub Issue: #42 | https://github.com/Chuseok22/chuseok22-home-web/issues/42 -->
    ❗[버그][인증] 카카오 로그인 실패 문제
    ...

GitHub 이슈 생성이 실패한 경우에는 이 줄을 추가하지 않습니다.

# 템플릿별 출력 형식

## 버그 이슈

```markdown
❗[버그][카테고리] 설명

🗒️ 설명
---

- {사용자 입력 기반으로 버그 현상 설명}

🔄 재현 방법
---

1. {재현 단계 1}
2. {재현 단계 2}
3. {결과 확인}

📸 참고 자료
---

{사용자가 로그나 스크린샷을 제공한 경우 여기에 정리, 없으면 비워둠}

✅ 예상 동작
---

- {정상적으로 동작해야 하는 모습}

⚙️ 환경 정보
---

- **OS**:
- **브라우저**:
- **기기**:

🙋‍♂️ 담당자
---

- **백엔드**: 이름
- **프론트엔드**: 이름
- **디자인**: 이름
```

긴급 버그인 경우 제목만 아래처럼 변경합니다.

```markdown
❗[긴급][버그][카테고리] 설명
```

---

## 기능추가 / 기능개선 / 기능요청 이슈

기능추가:

```markdown
⚙️[기능추가][카테고리] 설명
```

기능개선:

```markdown
🚀[기능개선][카테고리] 설명
```

기능요청:

```markdown
🔧[기능요청][카테고리] 설명
```

긴급인 경우:

```markdown
⚙️[긴급][기능추가][카테고리] 설명
🚀[긴급][기능개선][카테고리] 설명
🔧[긴급][기능요청][카테고리] 설명
```

본문:

```markdown
📝 현재 문제점
---

- {현재 어떤 부분이 부족하거나 필요한지}

🛠️ 해결 방안 / 제안 기능
---

- {제안하는 기능이나 해결 방향}

⚙️ 작업 내용
---

- {작업 항목 1}
- {작업 항목 2}

🙋‍♂️ 담당자
---

- **백엔드**: 이름
- **프론트엔드**: 이름
- **디자인**: 이름
```

주의:

- `⚙️ 작업 내용`은 기능추가 또는 기능개선에서만 필요할 때 추가합니다.
- 작업 내용은 코드나 구현 방식이 아니라 작업 단위로만 작성합니다.

---

## 디자인 요청 이슈

```markdown
🎨[디자인][카테고리] 설명

🖌️ 요청 내용
---

- {디자인 요청 내용}

🎯 기대 결과
---

- {디자인 적용 후 예상 결과}

📋 참고 자료
---

{참고 링크, 이미지 등}

💡 추가 요청 사항
---

- {추가 고려 사항}

🙋‍♂️ 담당자
---

- **백엔드**: 이름
- **프론트엔드**: 이름
- **디자인**: 이름
```

긴급 디자인 이슈인 경우 제목만 아래처럼 변경합니다.

```markdown
🎨[긴급][디자인][카테고리] 설명
```

---

## QA / 시험 요청 이슈

```markdown
🔍[시험요청][카테고리] 설명

🔗 ISSUE 정보
---

- {관련 이슈 번호 또는 링크}

🔗 PR 정보
---

- {관련 PR 번호 또는 링크}

🧩 시험 대상
---

- {테스트할 기능 또는 수정사항}

📋 테스트 시나리오
---

1. {테스트 케이스 1}
2. {테스트 케이스 2}
3. {테스트 케이스 3}

⚙️ 테스트 환경
---

- **프로젝트 Version**:
- **OS**:
```

# GitHub Repository 실제 이슈 생성 규칙

## 생성 순서

1. 프로젝트 root 기준으로 Repository를 추출합니다. (`git remote -v` 또는 `gh repo view`)
2. 이슈 타입과 카테고리를 자동 판단합니다.
3. 제목과 본문을 작성합니다.
4. `.issue/` 폴더에 이슈 파일을 저장합니다.
5. Repository의 기존 Label 목록을 조회합니다. (`gh api repos/{owner}/{repo}/labels`)
6. 사용할 수 있는 Label만 선별합니다.
7. 현재 GitHub 인증 사용자를 확인합니다. (`gh api user` 또는 `gh auth status`)
8. 확인된 현재 사용자가 있으면 Assignee로 지정합니다.
9. 실제 GitHub Repository에 이슈를 생성합니다. (MCP `create_issue` 또는 `gh issue create`)
10. 이슈 생성 성공 시 `.issue/` 파일 상단에 GitHub 이슈 번호와 URL을 기록합니다.
11. 이슈 생성 후에도 `.issue` 파일은 유지합니다.

> **참고**: 이슈 생성 후 GitHub Actions workflow가 자동으로 helper comment(브랜치명·커밋 메시지 템플릿 포함)를 이슈에 추가합니다. 이 스킬은 해당 comment를 별도로 작성하지 않습니다.

## GitHub 이슈 생성 조건

아래 정보가 있으면 실제 GitHub Repository에 이슈를 생성합니다.

- Repository 정보
- 이슈 제목
- 이슈 본문

아래 정보는 선택 사항입니다.

- Assignee
- Label

선택 정보가 없더라도 이슈 생성은 진행할 수 있습니다.

## Repository 추출 실패 시

Repository를 추출하지 못한 경우:

- `.issue/` 파일은 생성합니다.
- GitHub 이슈는 생성하지 않습니다.
- 최종 출력에 실패 사유를 명확히 표시합니다.

## Assignee 확인 실패 시

현재 GitHub 사용자를 확인하지 못한 경우:

- Assignee 없이 이슈를 생성합니다.
- 사용자에게 Assignee를 물어보지 않습니다.
- 최종 출력에 Assignee가 지정되지 않았음을 표시합니다.

## Label 조회 실패 시

Repository Label을 조회하지 못한 경우:

- Label 없이 이슈를 생성합니다.
- 새 Label을 만들지 않습니다.
- 최종 출력에 Label이 지정되지 않았음을 표시합니다.

## GitHub 이슈 생성 실패 시

실패한 경우 아래 내용을 출력합니다.

- 로컬 이슈 파일 저장 경로
- GitHub 생성 실패 이유
- Repository 정보
- 이슈 제목
- 필요한 추가 정보

성공하지 않았는데 성공했다고 말하지 않습니다.

# 최종 출력 형식

## GitHub 이슈 생성 성공 시

```markdown
## 저장 완료

- 파일 경로: `.issue/{파일명}` (GitHub 이슈 번호 기록 완료)

## GitHub 이슈 생성 결과

- Repository: `{owner/repository}`
- Issue URL: `{생성된 이슈 URL}`
- Assignee: `{현재 GitHub 사용자 또는 없음}`
- Labels: `{적용된 Label 목록 또는 없음}`

## 이슈 요약

- 제목: `{이슈 제목}`
- 타입: `{이슈 타입}`
- 카테고리: `{카테고리}`
- 긴급 여부: `{예/아니오}`

GitHub에 이슈가 생성되었습니다.
```

## GitHub 이슈 생성 실패 시

```markdown
## 저장 완료

- 파일 경로: `.issue/{파일명}`

## GitHub 이슈 생성 결과

- 생성하지 못했습니다.
- 사유: `{사유}`
- Repository: `{owner/repository 또는 확인 불가}`
- Assignee: `{현재 GitHub 사용자 또는 없음}`
- Labels: `{적용 예정이던 Label 목록 또는 없음}`

## 이슈 요약

- 제목: `{이슈 제목}`
- 타입: `{이슈 타입}`
- 카테고리: `{카테고리}`
- 긴급 여부: `{예/아니오}`

GitHub에 이슈를 생성하려면 위 제목과 본문을 복사해서 붙여넣으세요.
```

# 작성 원칙

1. 사용자의 입력이 짧아도 문맥상 명확하면 바로 이슈를 작성합니다.
2. 이슈 타입은 이슈의 성격에 따라 자동 지정합니다.
3. 이슈 타입과 카테고리가 불명확할 때만 질문합니다.
4. 긴급이 아닌 경우 제목의 대괄호는 반드시 2개만 사용합니다.
5. 긴급인 경우 제목의 대괄호는 반드시 3개만 사용합니다.
6. `[긴급]`은 사용자가 직접 긴급이라고 말한 경우에만 사용합니다.
7. 이슈 본문에는 코드, 구현 방법, 코드 예시를 작성하지 않습니다.
8. 본문 담당자 섹션은 템플릿 기본값인 `이름`을 유지합니다. (팀 프로젝트·개인 프로젝트 모두 동일하게 적용. 임의 수정 금지)
9. GitHub Assignee는 Claude Code를 실행하는 현재 사용자로 설정합니다.
10. 현재 GitHub 사용자를 확인할 수 없으면 Assignee를 지정하지 않습니다.
11. Assignee가 없다는 이유로 질문하지 않습니다.
12. Repository는 프로젝트 root 기준으로 추출합니다.
13. Repository Label은 read만 수행합니다.
14. 새 Label은 절대 생성하지 않습니다.
15. Repository에 존재하는 Label만 사용합니다.
16. 적절한 Label이 없으면 Label 없이 이슈를 생성합니다.
17. `.issue/`는 프로젝트 root에 생성합니다.
18. 이슈 파일은 항상 먼저 저장하고, 그다음 GitHub 이슈 생성을 시도합니다.
19. GitHub 이슈 생성 후에도 `.issue` 파일은 유지합니다.
20. GitHub 이슈 생성이 실패해도 로컬 파일 저장 결과는 유지합니다.
21. 실패 사유는 명확하게 설명합니다.
22. GitHub 이슈 생성 성공 시 `.issue/` 파일 상단에 이슈 번호와 URL을 기록합니다. 실패 시에는 기록하지 않습니다.

1. GitHub 이슈 타입 자동 판단
2. GitHub 이슈 제목 생성
3. GitHub 이슈 본문 생성
4. 프로젝트 root `.issue/` 저장 파일 생성
5. Repository 기존 Label 조회 후 적용
6. 현재 GitHub 사용자 Assignee 적용
7. 실제 GitHub Repository 이슈 생성
8. `.issue` 파일 유지
